3. evaluateTestCases()  (lib/sandbox.ts)
──────────────────────
  Function signature:
    evaluateTestCases(stdout: string, testCases: TestCase[])
      → { passed: number; total: number; results: TestCaseResult[] }

  TestCaseResult: { id, description, passed, hidden }

  Logic:
    For each TestCase:
      1. All outputContains strings must appear in stdout
      2. No outputNotContains string may appear in stdout
      3. outputMatchesPattern regex must match (if provided)
    All conditions must hold → passed: true

  Edge cases handled:
    • Invalid regex → test fails gracefully (try/catch)
    • Empty stdout → only "non-empty" tests fail, others pass/fail normally
    • null stdout → treated as ""

DESIGN DECISIONS
────────────────
1. Test cases are OUTPUT ASSERTIONS, not input injection.
   CUDA programs compile and run as a single binary — there is no stdin-based
   input injection without modifying the code. Instead, tests check what the
   program prints to stdout. This is the correct model for a learning sandbox.

2. Three test types cover the space:
   outputContains   — positive assertions ("result should say CORRECT")
   outputNotContains — negative assertions ("result must not say WRONG")
   outputMatchesPattern — structural assertions ("bandwidth should be a number")

3. Hidden tests are revealed on Submit, not Run.
   The last 8–10 tests per project are marked hidden:true. Their descriptions
   are hidden from the learner during Run (shown as "hidden test") but revealed
   after Submit, giving Submit genuine grading value.

4. evaluateTestCases is pure client-side.
   No server round-trip needed for test evaluation — the sandbox already
   returns stdout. The evaluate function runs instantly in the browser.
